# A 12-node satellite mesh in 30 seconds

A real packet network is not flat. The latency of a hop depends on
how far you're going, what link you're on, and how loaded that
link is. So why are most network labs flat — every node–pair
seeing the same delay, the same loss, the same bandwidth?

This recipe builds an Iridium-style 12-satellite ring with three
distinct path classes:

- **Ring neighbors** (12 short-haul links): low delay, low loss.
- **Cross-plane links** (3 long-haul jumps across the ring): high
  delay, moderate loss, capped bandwidth.
- **One degraded link** (operating on a bad antenna): asymmetric
  jitter, heavy loss, severely capped throughput.

All 12 nodes share **one L2 segment**. Each pair gets its own
delay/loss profile via the kernel's HTB+netem+flower TC primitives,
applied per-source-interface via netlink.

There is no way to express this in a single containerlab or
Docker-Compose topology. Docker's bridge network gives every
container the same egress queue — there's no per-destination netem
on a shared L2.

## The full topology

[`examples/cookbook/satellite-mesh.nll`](../../examples/cookbook/satellite-mesh.nll):

```nll
lab "satellite-mesh" {
  description "12-node Iridium-style ring mesh"
  tags [cookbook, mesh, satellite]
}

for i in 0..11 {
  node sat${i}
}

network mesh {
  members [
    sat0:rf,  sat1:rf,  sat2:rf,  sat3:rf,
    sat4:rf,  sat5:rf,  sat6:rf,  sat7:rf,
    sat8:rf,  sat9:rf,  sat10:rf, sat11:rf,
  ]
  subnet 172.100.0.0/24

  # Ring neighbors — 12 short-haul links generated by a for-loop
  # using modulo arithmetic. Both directions of every pair.
  for i in 0..11 {
    impair sat${i} -- sat${(i + 1) % 12} { delay 12ms jitter 2ms loss 0.1% }
    impair sat${(i + 1) % 12} -- sat${i} { delay 12ms jitter 2ms loss 0.1% }
  }

  # Cross-plane links — 3 long-haul jumps with rate caps.
  for i in 0..2 {
    impair sat${i} -- sat${i + 6} { delay 80ms jitter 15ms loss 2% rate-cap 50mbit }
    impair sat${i + 6} -- sat${i} { delay 80ms jitter 15ms loss 2% rate-cap 50mbit }
  }

  # One degraded antenna.
  impair sat3 -- sat9 { delay 95ms jitter 40ms loss 8% rate-cap 5mbit }
  impair sat9 -- sat3 { delay 95ms jitter 40ms loss 8% rate-cap 5mbit }
}
```

That's the entire definition. 32 directional impair rules from
~25 lines of NLL.

## How the for-loop expands

The `for i in 0..11 { … impair sat${i} -- sat${(i + 1) % 12} … }`
block runs at parse time. NLL evaluates `${(i + 1) % 12}` with full
arithmetic precedence and modulo, then materializes one
`NetworkImpairDef` per iteration. You can see the expansion with
`render`:

```bash
$ nlink-lab render examples/cookbook/satellite-mesh.nll | grep impair | head
  impair sat0 -- sat1 { delay 12ms jitter 2ms loss 0.1% }
  impair sat1 -- sat0 { delay 12ms jitter 2ms loss 0.1% }
  impair sat1 -- sat2 { delay 12ms jitter 2ms loss 0.1% }
  ...
  impair sat0 -- sat6 { delay 80ms jitter 15ms loss 2% rate-cap 50mbit }
  impair sat6 -- sat0 { delay 80ms jitter 15ms loss 2% rate-cap 50mbit }
  ...
```

`for` works at top-level (to generate nodes/links) and inside
`network { }`, `node { }`, and `nat { }` blocks. Loop variables
substitute via `${var}`, and any `${expr}` containing `+`, `-`,
`*`, `/`, `%`, parentheses, or ternaries (`a == b ? x : y`)
evaluates at parse time.

## Run it

```bash
sudo nlink-lab deploy examples/cookbook/satellite-mesh.nll
```

The deploy creates 12 namespaces, 12 veth pairs, one bridge, and
12 per-source TC trees (one per node, each with up to 4
per-destination netem leaves). It takes well under a second on a
laptop.

Now ping between nodes and watch the latency match the topology:

```bash
# Ring-adjacent: ~12ms
sudo nlink-lab exec satellite-mesh sat0 -- ping -c 3 172.100.0.2

# Cross-plane jump: ~80ms
sudo nlink-lab exec satellite-mesh sat0 -- ping -c 3 172.100.0.7

# Degraded antenna: ~95ms with packet loss
sudo nlink-lab exec satellite-mesh sat3 -- ping -c 10 172.100.0.10
```

## What the kernel actually built

For each source node, nlink-lab built one
[`PerPeerImpairer`](https://docs.rs/nlink/latest/nlink/netlink/impair/struct.PerPeerImpairer.html)
tree, which materializes as:

```text
sat0:rf
└── 1: htb (root)
    └── 1:1 htb (parent — sums child rates)
        ├── 1:2 htb -- 10: netem (delay 12ms loss 0.1%)  ← flower(dst=sat1) classid=1:2 prio=100
        ├── 1:3 htb -- 11: netem (delay 12ms loss 0.1%)  ← flower(dst=sat11) classid=1:3 prio=101
        ├── 1:4 htb -- 12: netem (delay 80ms loss 2%)    ← flower(dst=sat6) classid=1:4 prio=102
        └── 1:5 htb (default, no netem leaf)
```

You can inspect any of these directly:

```bash
sudo nlink-lab exec satellite-mesh sat0 -- tc -s qdisc show dev rf
sudo nlink-lab exec satellite-mesh sat0 -- tc -s class show dev rf
sudo nlink-lab exec satellite-mesh sat0 -- tc filter show dev rf
```

This is the standard kernel TC stack, applied directly. nlink-lab
adds no kernel modules, no userspace shaper daemon — all of it is
the Linux kernel doing what it was already capable of.

## Tear down

```bash
sudo nlink-lab destroy satellite-mesh
```

## Variations

- **Asymmetric impairment.** Drop one of each pair to model a
  one-direction link issue. The `impair sat3 -- sat9` rule applies
  to traffic *from* sat3 *to* sat9, not the reverse.
- **Add a partition mid-test.** Use a `scenario { at 5s { down sat3:rf } at 15s { up sat3:rf } }` block.
- **Per-pair rate caps without delay.** Drop the `delay` token and
  keep only `rate-cap` for pure throughput shaping.
- **Reconcile, don't redeploy.** `nlink-lab apply
  examples/cookbook/satellite-mesh.nll` (after editing) will
  converge live state to match the new file with zero packet loss
  on unchanged links.

## When this is the right tool

- Testing a P2P/mesh protocol's behavior under realistic
  per-link conditions.
- Satellite or vehicular networking software that's sensitive to
  per-hop latency variance.
- Embedded routing software that you'd like to fuzz across
  topology classes.

## When it isn't

If you need a Cisco/Nokia/Juniper NOS image, use
[containerlab](https://containerlab.dev). nlink-lab cannot run a
vendor container.
